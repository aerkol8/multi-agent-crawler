For production deployment, run the crawler as a supervised service (systemd, Kubernetes job, or Nomad task) with SQLite on durable SSD storage and WAL enabled. Keep the current architecture as a single-machine service but separate concerns operationally: one long-running index worker process, one read-only search API/CLI process, and a lightweight status endpoint. Add structured logging, heartbeat monitoring, disk-usage alerts, and periodic checkpoint validation. The current queue-depth cap, worker pool, and global rate limiter already provide good load control; production hardening should focus on observability, retry policies, and safe shutdown/restart guarantees.

As scale grows, evolve this design incrementally rather than rewriting. First, partition by crawl run and domain to reduce contention and improve recovery windows. Next, move from direct SQLite term-table writes to segment files plus periodic merges to reduce write amplification. If single-machine limits are reached, split roles into separate services (fetch/discovery, indexing, query) with a shared durable queue while preserving the same contracts: dedup invariants, depth metadata, and tuple-based query output. This keeps migration risk low and lets you retain the current CLI and quality checks during transition.
