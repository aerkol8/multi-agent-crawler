For production deployment, run the crawler as a supervised localhost service (systemd, Kubernetes, or Nomad) with SQLite WAL on SSD and explicit operational thresholds: alert when disk usage exceeds 75 percent, force maintenance mode at 90 percent, emit a heartbeat every 10 seconds, and run WAL checkpoint validation every 60 seconds or 5,000 discovered URLs (whichever comes first). Keep indexing and querying as separate processes on the same host, enforce global RPS and queue-depth limits, and use graceful shutdown with a 10-second drain window so resume can continue without duplicate crawling. Configure retries with exponential backoff (0.2s, 0.4s, 0.8s) and cap consecutive host failures before temporary skip to protect crawl throughput.

Scale this architecture incrementally using measurable triggers rather than a rewrite: if crawl database size exceeds 80 GB, p95 search latency exceeds 300 ms for 24 hours, or sustained discovery falls below 30 URLs/sec at target RPS, move first to run/domain partitioning and then to segment-based index writes with periodic merges to reduce write amplification. Only after those steps should you split fetch/discovery, indexing, and query into separate services with a shared durable queue, while preserving the same contracts (URL dedup invariants, depth metadata, and tuple-only search output) so existing CLI/web workflows and grading checks remain valid during migration.
